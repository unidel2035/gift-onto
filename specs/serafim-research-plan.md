# План исследований: Серафим как ядро проекта «Звездочёт-Серафим»

## Исходное состояние

Серафим V2 Q8 — квантованная LLM (Qwen2-1.5B, 1.6 ГБ). 5/5 правильных тактических решений
на статической текстовой выборке. **К полётам не готов.**

## Что мы делаем по гранту (12 месяцев, 5 человек)

Не делаем дрон. Не делаем планер. Делаем **среду обучения социального поведения агентов**
и доводим Серафима до состояния «можно ставить на борт».

---

## Направление 1: Интеграция Серафима в симуляционную среду

### 1.1. Serafim-in-the-Loop Simulation
Заменить хардкодные правила в training_arena.py на реальный инференс Серафима.

**Что сделать:**
- API-мост между симуляцией и Ollama (локально) / llama.cpp (на Orange Pi)
- Формат входа: структурированное описание тактической ситуации (JSON → текст)
- Формат выхода: действие + цель + параметры (парсинг ответа Серафима)
- Пакетная обработка: до 10 агентов × 10 Гц = 100 запросов/с

**Риски:** задержка инференса на CPU (400 мс) может быть критична для real-time симуляции.
**Решение:** асинхронный инференс + prediction (агент действует по предыдущему решению пока ждёт новое).

### 1.2. Сенсорный интерфейс
Серафим должен воспринимать не текст, а данные с сенсоров.

**Форматы входа:**
| Сенсор | Что передаём | Размер |
|--------|-------------|--------|
| YOLO-детектор | Список обнаруженных объектов: класс, дистанция, азимут, уверенность | ~200 байт |
| UWB-сетка | Положение соседей в рое: ID, X, Y, Z, статус | ~100 байт |
| IMU | Курс, тангаж, крен, скорость | ~50 байт |
| Батарея | Остаток %, потребление Вт | ~10 байт |
| LoRa RSSI | Качество связи с каждым соседом | ~50 байт |
| РЭБ-детектор | Уровень помех по диапазонам | ~30 байт |

**Формат выхода:**
```
ACTION: <attack|observe|rtb|patrol|support>
TARGET: <target_id|direction|none>
PRIORITY: <0-10>
REASON: <краткое обоснование>
```

### 1.3. Коннектор к полётному контроллеру
Перевод тактического решения в команды MAVLink/ArduPilot.

**Цепочка:** Серафим → действие → FlightController.mission → MAVLink-команды → БпЛА

## Направление 2: Среда социального поведения агентов

Это **главное исследовательское направление**. В существующих симуляциях агенты действуют по правилам.
Мы заменяем правила на LLM-агентов и наблюдаем **эмерджентное поведение**.

### 2.1. Базовые социальные сценарии

| Сценарий | Агентов | Вопрос исследования |
|----------|---------|-------------------|
| **Одиночка** | 1 | Базовое тактическое поведение без coordination |
| **Пара** | 2 | Возникает ли cooperation? Разделение ролей? |
| **Тройка** | 3 | Распределение целей. Аукцион. Лидерство. |
| **Рой (5-10)** | 5-10 | Масштабирование. Конфликты. Consensus. |
| **Два роя** | 2×(3-5) | Межроевое взаимодействие. Конкуренция. |
| **Рой + РЭБ** | 5 | Поведение при потере связи. Автономность. |
| **Рой + ПВО** | 5 | Уклонение. Жертвование. Приоритезация угроз. |

### 2.2. Протокол взаимодействия агентов

Агенты обмениваются сообщениями через LoRa-пакеты ≤51 байт.

**Формат сообщения:**
```
[1B type][1B sender][1B priority][4B target_coords][44B payload]
```

**Типы сообщений:**
- `TARGET_FOUND` — я обнаружил цель, координаты
- `TARGET_CLAIMED` — я беру эту цель на себя
- `NEED_SUPPORT` — нужна помощь (РЭБ, ПВО, сложная цель)
- `RTB` — ухожу на базу (низкая батарея, повреждение)
- `STATUS` — я жив, моя позиция, моя задача
- `CONSENSUS_VOTE` — голосование за план роя

### 2.3. Метрики социального поведения

| Метрика | Что измеряет |
|---------|-------------|
| **Cooperation Index** | Доля совместных действий от всех действий |
| **Conflict Rate** | Доля конфликтов (два агента выбрали одну цель) |
| **Consensus Time** | Время до консенсуса роя (мс) |
| **Role Distribution** | Энтропия распределения ролей (разведчик/удар/ретранслятор) |
| **Survival Rate** | Доля агентов выполнивших миссию и вернувшихся |
| **Mission Success** | Доля поражённых целей от заданных |
| **Communication Efficiency** | Байт на одно тактическое решение |
| **Emergent Complexity** | Количество непредусмотренных (не запрограммированных) поведений |

### 2.4. Эмерджентное поведение — ключевой результат

Мы **не программируем** роевое поведение. Мы создаём среду, в которой LLM-агенты **самоорганизуются**.

**Что мы ищем:**
- Спонтанное разделение на разведчиков и ударников
- Возникновение ситуационного лидерства
- Невербальная координация (агент А видит что Б атакует → поддерживает)
- Самопожертвование (камикадзе прикрывает разведчика)
- Адаптация к новой тактике противника без перепрограммирования

**Метод:** запускаем 100+ симуляций с разными начальными условиями, записываем все действия,
кластеризуем паттерны поведения, выделяем emergent phenomena.

## Направление 3: Генерация обучающих данных

### 3.1. Синтетический датасет из симуляций

Симуляция → запись всех состояний и решений → датасет для дообучения.

**Объём:** 10 000 сценариев × 10 агентов × 100 шагов = 10 млн примеров.

**Формат:** тот же ChatML что использовался для V2/V3.

**Классы сценариев:**
- По числу агентов: 1, 2, 3, 5, 10
- По типу целей: статические, движущиеся, защищённые ПВО, под РЭБ
- По местности: открытая, городская, лесная, горная
- По времени: день, ночь, сумерки
- По погоде: ясно, дождь, туман, ветер

### 3.2. Дообучение на симулированных данных

LoRA (r=16, alpha=32, 18.4M параметров) на 10 000 примерах.
Ожидаемое улучшение: точность с 5/5 на статике → 95%+ на динамических сценариях.

### 3.3. Данные из реальных полётов (апробация)

Во время этапа апробации (месяцы 11-12): запись реальных полётных данных → дообучение.

## Направление 4: Оценка и верификация

### 4.1. Бенчмарк Serafim-Bench

Стандартизированный набор тестов:

| Тест | Описание | Метрика |
|------|----------|--------|
| SingleTarget | Одна цель, разная дистанция, разная батарея | Accuracy |
| MultiTarget | 3-5 целей разного приоритета | Priority F1 |
| LostComms | Обрыв связи на 10-60 секунд | Recovery Rate |
| LowBattery | Батарея 5-20%, разные дистанции до цели и базы | RTB Accuracy |
| EWEnvironment | РЭБ-подавление, зашумлённые сенсоры | Stability |
| Adversarial | Противник использует контр-тактику | Adaptation |
| FriendlyFire | Свой БпЛА в зоне поражения | Safety (должен быть 0%) |
| Civilian | Гражданский объект в кадре | Safety (должен быть 0%) |

### 4.2. Safety Verification

**Категорические требования (блокируют полёт при нарушении):**
- Никогда не атаковать свои БпЛА (IFF)
- Никогда не атаковать гражданские объекты
- Не лететь в землю (terrain avoidance)
- Не выходить за пределы зоны миссии

**Метод:** формальная верификация через перебор граничных случаев + adversarial testing.

## Направление 5: Аппаратная оптимизация

### 5.1. Бенчмарк на целевом железе

| Платформа | ОЗУ | NPU/GPU | Ожидаемая задержка |
|-----------|-----|---------|-------------------|
| Orange Pi 5 | 8 ГБ | Rockchip NPU 6 TOPS | ~50 мс (NPU) / ~400 мс (CPU) |
| Orange Pi 5 Plus | 16 ГБ | Rockchip NPU 6 TOPS | ~50 мс (NPU) |
| Raspberry Pi 5 | 8 ГБ | Нет | ~600 мс (CPU) |
| Jetson Orin Nano | 8 ГБ | 70 TOPS | ~15 мс (GPU) |

### 5.2. Оптимизация инференса

- Квантование Q4_K_M (~500 МБ) vs Q8_0 (~1.6 ГБ) — сравнение точность/скорость
- NPU-офлоад: YOLO на NPU, Serafim на CPU параллельно
- Пакетный инференс для роя (один запрос — решения для всех)
- Предварительная загрузка модели в RAM (исключить холодный старт)

## Направление 6: Исследовательские вопросы

Это научная часть гранта — вопросы, на которые нет ответа в литературе:

1. **Emergent coordination**: возникает ли cooperation между LLM-агентами без явного программирования?
2. **Scalability ceiling**: при каком размере роя LLM-координация перестаёт работать?
3. **Adversarial robustness**: насколько LLM-агент устойчив к adversarial-примерам в сенсорных данных?
4. **Transfer learning**: переносится ли тактика из симуляции в реальный полёт?
5. **Explainability**: можем ли мы объяснить почему Serafim принял конкретное решение?
6. **Ethics of autonomous agents**: при каких условиях автономный агент принимает решение о lethal force?

## Что НЕ входит в грант (не делаем)

- Разработка планера БпЛА (аэродинамика, материаловедение)
- Разработка транспортно-пусковой установки
- Разработка полётного контроллера (используем COTS: ArduPilot/PX4)
- Разработка камеры или оптической системы
- Разработка полезной нагрузки (боевая часть)
- Сертификация и войсковые испытания
